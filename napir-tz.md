# Напір — ТЗ (v2)

Веб-калькулятор тиску води в приватному будинку зі свердловиною. Клієнтський застосунок без бекенду, придатний для хостингу на GitLab / GitHub Pages.

> Версія 2 ТЗ. Зміни відносно v1: **модель схеми стала графом, а введення — візуальним редактором (§4.0)**; тип джерела став явним вибором і керує моделлю (§3.0, §3.2, §4); переобґрунтовано вибір формули тертя й запас на фітинги внесено в саму модель (§3.1); чесно зафіксовано межу прямого режиму до кривої насоса (§3.3); додано перевірку швидкості потоку (§3.4); цільовий тиск задається через пресети приладів (§4.1); переписано критерій верифікації (§5); зафіксовано стек (§6).

## 1. Навіщо

Підбирати обладнання водопостачання (насос, труби) і розуміти, який тиск буде в крані — без EPANET та інженерних пакетів. Інструмент рахує чесну гідравліку, але ховає зайву складність за простим UI під один сценарій: джерело → станція/бустер → стояк → поверхи.

Прототип (`napir.html` v0.1) реалізує базу для сценарію «поверхнева станція». Це ТЗ фіксує, що є, і куди рости.

## 2. Кого обслуговує

Власник будинку, який щось розуміє в трубах, але не інженер-гідравлік. Мова інтерфейсу — українська. Одиниці: метри, бари, л/хв, мм.

## 3. Фізична модель

### 3.0. Тип джерела (нове — керує всією моделлю)

Перший вибір користувача. Визначає формулу статичного підйому й набір попереджень.

| Тип | Опис | Ліміт всмоктування | Статичний підйом |
|---|---|---|---|
| **Поверхнева станція** | Насос над водою, всмоктує з глибини (свердловина/колодязь неглибокі) | Так, ~7 м | `depth_water + Σ rise_i` |
| **Бак + бустер** | Накопичувальний бак у будинку + підвищувальний насос. Для глибоких/слабких свердловин | Немає (бак поруч із насосом) | `Σ rise_i` (глибина до бака не всмоктується) |

**Поза обсягом (свідомо):** заглибний насос у свердловині. Для нього глибина стає частиною напору, що *качається* (не всмоктується), ліміту 7 м немає, а «напір станції» замінюється характеристикою заглибного насоса. Не моделюється у v1, щоб не роздувати обсяг; винесено сюди явно, щоб назва інструмента не вводила в оману.

### 3.1. Втрати на тертя — Хазен–Вільямс

```
h_f = 10.67 · L_eff · Q^1.852 / (C^1.852 · D^4.87)   [м]
```

- `L_eff` — **ефективна** довжина ділянки, м = `L_geom · (1 + k_fittings)` (див. нижче)
- `Q` — витрата, м³/с (у UI вводиться в л/хв, конвертується)
- `D` — внутрішній діаметр, м (у UI — мм)
- `C` — коефіцієнт шорсткості (пластик 150, оцинковка нова 120, сталь стара 100)

**Чому саме Хазен–Вільямс, а не Дарсі–Вейсбах.** Для робочого діапазону інструмента (труби 13–26 мм, пластик C≈150, вода кімнатна, 5–15 л/хв) число Рейнольдса ~16 000 — розвинений турбулентний режим, у якому HW і DW сходяться в межах ~10–15% (на тонких ½" трубах виміряно ~12% — HW дещо занижує; див. тести Етапу 1). Головні слабкості HW (холодна вода, ламінарний режим) тут не спрацьовують. Водночас домінантна похибка моделі — це проігноровані локальні опори фітингів (легко +40% тертя на стояку). Тому міняти двигун заради ~5% немає сенсу, поки не закрито фітинги. DW реалізується як **незалежний тест-оракул** (§5), а не як двигун.

**Запас на фітинги (нове, у самій моделі — не в беклозі).** Кожна ділянка має коефіцієнт `k_fittings` (частка додаткової еквівалентної довжини), за замовчуванням ~0.2–0.3 залежно від типу ділянки (стояк з колінами — більше, прямий магістральний прогін — менше). Прибирає систематичний оптимізм результату. Пізніше може уточнитися до покоефіцієнтного розрахунку по кожному фітингу (беклог).

### 3.2. Статичний підйом

**Через висоти вузлів (нове — вмикається графом, §4.0).** У графі кожен вузол має висоту `z` (свердловина/дзеркало — від'ємна, ввід у будинок — 0, кран 2 поверху — додатна). Підйом труби виводиться як різниця висот її кінців, а не вводиться руками на кожній ділянці. Це прибирає задвоєння й помилки ручного поля `rise`.

Статичний підйом уздовж активного шляху від джерела до крана:

```
static_lift = z_tap − z_source            // загальна форма через висоти вузлів
```

Для сумісності з термінологією v1:

```
static_lift = depth_water + Σ rise_i      // поверхнева станція (розкладка через ділянки)
static_lift =               Σ rise_i      // бак + бустер (дзеркало не всмоктується)
```

- `z_source` — висота джерела: для поверхневої станції дно = `−depth_water`; для «бак+бустер» = висота бака (зазвичай ~0).
- `z_tap` — висота крана/приладу.

### 3.3. Баланс напору

Тиск у точці розбору:

```
head_at_tap = H_pump(Q) − static_lift − friction_total   [м]
P_bar = head_at_tap / 10.2
```

Напір насоса тепер залежить від витрати через **криву насоса** (реалізовано, див. §3.5), а не задається константою.

Зворотна задача (потрібний напір під заданий тиск):

```
H_pump_required = P_target · 10.2 + static_lift + friction_total
```

Це — напір, потрібний **при заданій витраті** `Q`; паспортний макс. напір станції буде вищим (він при нульовій подачі).

Перевідник: **1 бар = 10.2 м** напору.

**Що лишається спрощенням (чесно).** Витрата `Q` досі задається користувачем (через прилад або вручну), а не розв'язується з положення крана. Тобто рахуємо тиск, який станція дасть *при відкритому крані на витрату Q*; повністю відкритий кран пропустив би більшу витрату й дав менший тиск. Це прийнятно для оцінки однієї точки; повний розвʼязок робочої точки крана — поза обсягом v1.

### 3.5. Крива насоса (реалізовано, Етап 2.5)

Насос дає не один напір, а криву. Мінімальна модель на два паспортні числа з коробки станції — макс. напір `H_max` (при нульовій подачі) і макс. подача `Q_max` (при нульовому напорі):

```
H_pump(Q) = H_max · (1 − (Q / Q_max)²)   [м]
```

Це прибирає найгірший інпут v0.1 — «напір у робочій точці», якого на паспорті немає. Користувач вводить два числа, які реально надруковані на станції, а робочу точку рушій рахує сам. Уточнення на 3 точки з графіка паспорта — можливе, але не обовʼязкове.

### 3.6. Гідроакумулятор — сайзинг (реалізовано, Етап 2.6)

Ортогонально до тиску: бак **не міняє усталений тиск** у крані, а визначає запас води й частоту пусків насоса. Окрема гілка калькулятора.

- Корисний запас (закон Бойля, тиски абсолютні): `V_запас = V_бака · P_перед·(1/P_увімк − 1/P_вимк)`.
- Передзарядка за замовчуванням: `P_увімк − 0.14 бар`.
- Макс. пусків/год (найгірший розбір = ½ подачі): `N = 15 · Q_насоса / V_запас`; зворотна задача дає потрібний об'єм бака.
- **Подача насоса в бак** береться з кривої (§3.5) при напорі = `тиск вимкнення + глибина до води` (насос спершу піднімає воду з глибини, потім тисне в бак). Тертя фідера знехтуване (малий член). Отже глибина реально впливає на наповнення, і спливає пастка: якщо `H_max < тиск_вимк·10.2 + глибина`, станція не дотягує до тиску вимкнення, реле не спрацьовує (попередження). Напр. станція 43 м із 6 м тисне в бак максимум ~3.6 бар, не 4.2.

### 3.4. Обмеження, які треба сигналізувати

- **Всмоктування (лише поверхнева станція):** дзеркало глибше ~7 м → насос фізично не всмокче; попередити й запропонувати «бак + бустер». Для типу «бак + бустер» це попередження не показується.
- **Від'ємний тиск у крані** → «води не дійде».
- **Занадто високий потрібний напір (>60 м)** → натяк, що труби завузькі, а не насос слабкий.
- **Швидкість потоку (нове):** `v = Q / (π·D²/4)`. Якщо `v` перевищує ~2 м/с — попередити «труба завузька для цієї витрати: шум, гідроудар, зайві втрати». Дешевий чек, заразом пояснює, чому напір з'їдається.

## 4. Функціональні вимоги

### 4.0. Модель схеми та візуальний редактор (нове — ядро v1)

Замість лінійного списку ділянок користувач будує схему на **canvas-редакторі**: накидає елементи, з'єднує трубами, працює з отриманим графом.

**Граф.**
- **Вузли** (елементи): джерело (свердловина+станція / бак+бустер), трійник (розгалуження), кран/прилад. Кожен вузол має висоту `z` (§3.2).
- **Ребра** (труби): несуть усі фізичні параметри ділянки — довжина `L`, внутрішній Ø, матеріал (пресет → C), `k_fittings`. Підйом *не* вводиться — виводиться з висот кінцевих вузлів.
- Список ділянок з v0.1 стає **виродженим випадком** графа (лінійний ланцюг), тож фізичні параметри нічого не втрачають — лише переїжджають на ребра.

**Топологічні обмеження v1** (щоб солвер лишався простим):
- Рівно **одне джерело**.
- **Дерево, без кілець** — гарантує унікальний шлях від джерела до кожного крана. Кільцеві магістралі → мережевий солвер, у беклог.

**Розрахунок — один активний кран за раз.** Користувач обирає кран на схемі → рахуються втрати вздовж єдиного шляху джерело→кран (той самий простий розрахунок, що й у v0.1, лише шлях береться з графа, а не зі списку). Одночасний розбір кількох кранів (розподіл потоку в трійниках, ітеративний мережевий солвер) — окремий етап беклогу, не блокує редактор.

**Редагування:** додати/видалити/перемістити вузол, провести/розірвати трубу, редагувати параметри вибраного елемента в боковій панелі. Схема — джерело істини стану (серіалізується в URL, беклог).

### 4.1. Два режими (є в v0.1)

- **Прямий** — «який тиск отримаю»: вхід — прилад + два паспортні числа станції (макс. напір, макс. подача), вихід — бари в крані + вердикт + напір насоса в робочій точці.
- **Зворотний** — «який насос потрібен»: вхід — прилад + бажаний тиск, вихід — потрібний напір при цій витраті.

**Пресети приладів у обох режимах (Етап 2.5).** Замість голого числа витрати — вибір приладу, що підставляє типову витрату (а у зворотному — ще й тиск). Користувач не знає витрати в жодному з режимів, тож пікер спільний:

| Прилад | ~тиск | ~витрата |
|---|---|---|
| Душ (комфорт) | 1.5 бар | 8 л/хв |
| Гігієнічний душ / змішувач | 1.0 бар | 6 л/хв |
| Пральна/посудомийна машина | 1.0 бар | 10 л/хв |
| Полив / шланг | 2.0 бар | 15 л/хв |
| Свій варіант | ручний ввід | ручний ввід |

Юзер думає «щоб душ був нормальний», а не «мені треба 2.5 бар». Пресет заразом підставляє витрату й частково знімає проблему подвійного вводу (§3.3).

### 4.2. Введення (є в v0.1, розширюється)

- Тип джерела (§3.0) — задається на вузлі-джерелі.
- Глибина дзеркала (лише для станції), витрата розбору / прилад, напір станції / бажаний тиск.
- **Схема-граф (§4.0)** — основний спосіб задати топологію. Параметри вибраного елемента (труби/вузла) редагуються в боковій панелі: назва, довжина, матеріал (пресет → Ø + C), внутрішній Ø, `k_fittings`, висота вузла.

### 4.3. Виведення (є в v0.1, розширюється)

- Головна цифра (бари або метри) + кольоровий вердикт.
- Розбивка: статичний підйом, сумарне тертя, втрати по кожній ділянці.
- Індикатор-смужка тиску, попередження (всмоктування / від'ємний тиск / швидкість / завузькі труби) відповідно до типу джерела.

### 4.4. Беклог (за пріоритетом)

1. ~~**Крива насоса замість числа.**~~ **Реалізовано (Етап 2.5)** у мінімальній 2-точковій формі (§3.5): паспортні макс. напір + макс. подача → напір у робочій точці рахується сам. Можливе уточнення до 3 точок з графіка паспорта.
2. **Покоефіцієнтні фітинги.** Уточнення §3.1: коліна, трійники, вентилі через ζ або точну еквівалентну довжину замість грубого множника `k_fittings`.
3. **Режим VFD-насоса** (Grundfos Scala2, DAB E.sybox): «тримає задані X бар, поки вистачає витрати». Джерело постійного тиску з обмеженням по Q. Природно лягає поверх кривої насоса (#1).
4. **Одночасний розбір кількох точок** (душ + кухня): розподіл потоку в трійниках, ітеративний мережевий солвер (Гарді-Кросс). Дерево-граф уже є (§4.0) — бракує лише розв'язувача розподілу витрат. *Найважчий пункт беклогу.*
5. **Збереження конфігурації в URL** — серіалізація графа в query-параметр, щоб ділитися посиланням «ось мій будинок».
6. **Порівняння сценаріїв поруч** — напр. стояк ½" vs ¾": дві колонки результатів.
7. ~~**Гідроакумулятор / бак-накопичувач.**~~ **Реалізовано (Етап 2.6)** — сайзинг об'єму бака й частоти пусків (§3.6). Далі можливо: буферизація повільної свердловини (скільки води віддасть до вичерпання).
8. **Заглибний насос** як третій тип джерела (§3.0) — інша формула статичного підйому, без ліміту всмоктування.
9. **Кільцеві магістралі** — зняти обмеження «дерево» (§4.0), розв'язувати граф із циклами. Разом з #4.

## 5. Нефункціональні вимоги

- **Без бекенду.** Уся логіка в браузері. Хостинг — статичні Pages, деплой без сервера.
- **Статичний артефакт після збірки.** Візуальний редактор (§4.0) вимагає фреймворка + збірки — вимога «один `index.html`» з v1 знімається. Натомість тверда межа: результат `vite build` — це статичний `dist/`, який лягає на GitHub/GitLab Pages **без сервера**. Залежності мінімальні й доцільні (див. §6).
- **Розрахунковий рушій відокремлений від UI** — чисті функції (`headloss`, `staticLift`, `solvePath`, `compute`) над структурою графа, щоб покривались тестами й переносились у CLI. Формула тертя — за інтерфейсом чистої функції, щоб DW-оракул можна було підмінити без переробки UI. Рушій не залежить від Svelte Flow — той лише редагує граф і віддає його рушію.
- **Верифікація (переписано).** Критерій «звірити з EPANET» доводить лише *правильність арифметики* — EPANET рахує ту саму формулу Хазена–Вільямса, тож збіг гарантований і мало що значить. Тому:
  - **Арифметика:** звірити 3+ сценарії з EPANET (HW) — розбіжність у межах округлення.
  - **Фізика:** незалежно реалізувати Дарсі–Вейсбах (з ν(T) і коеф. тертя за Swamee–Jain) як тест-оракул і звірити ті самі сценарії — спостережувана розбіжність HW↔DW ~10–15% у робочому діапазоні (HW занижує на тонких трубах). Ширша розбіжність = сигнал, що вийшли за межі застосовності HW. *Реалізовано на Етапі 1: `darcyWeisbach` + тест звірки.*
  - **Реальність (бажано):** хоч один польовий замір манометром на відомій конфігурації.
- **Адаптивність** — працює на телефоні (розбір у полі біля свердловини).
- **Доступність** — фокус з клавіатури, коректні `label`, розумний контраст.

## 6. Технічний стек (зафіксовано для v1)

- **Svelte + Svelte Flow + Vite.** Візуальний редактор графа (§4.0) будуємо на Svelte Flow — легша за React Flow, менше рантайму й церемоній, реактивність робить цикл «змінив граф → перерахував» майже безкоштовним. React Flow виграє лише популярністю, що для проєкту цього масштабу не окуповує ваги.
- **Розрахунковий рушій — чистий TS**, окремий модуль, незалежний від UI та Svelte Flow. HW-двигун + DW-оракул — окремі чисті функції. Юніт-тести — **Vitest**.
- **Деплой — статика без сервера.** `vite build` → `dist/`, публікується на Pages. Для GitHub Pages: виставити `base` у `vite.config` під ім'я репозиторію (project pages) + GitHub Actions, що білдить і публікує `dist/`. GitLab CI — аналогічний job. Сервера не треба.
- v0.1 (`napir.html`) — ванільний прототип; його фізику й пресети переносимо в рушій при рефакторі.

## 8. Приклад для перевірки

Дефолтний будинок (свердловина 10 м від дому, стояк ½" на 2 поверх), тип джерела — **поверхнева станція**:

| Параметр | Значення |
|---|---|
| Дзеркало води | 6 м нижче станції |
| Витрата | 10 л/хв |
| Всмоктування | ПНД 32 (внутр. 26), 12 м |
| Станція → ввід | ПНД 32 (внутр. 26), 10 м |
| Стояк на 2 поверх | ППР ½" (13 мм), 7 м, підйом 7 м |
| Гілка до змішувача | ППР ½" (13 мм), 4 м |

Прямий режим при напорі станції 25 м покаже оцінку тиску в душі 2 поверху (при заданій витраті 10 л/хв). Мета інструмента — щоб цю таблицю можна було заповнити реальними замірами й одразу побачити результат. Цей самий сценарій — основний кейс для верифікації §5.
