# Напір — план впровадження v1

План реалізації ТЗ v2 ([napir-tz.md](napir-tz.md)). Стек зафіксовано: **Svelte + Svelte Flow + Vite**, рушій — чистий TS, деплой — статичний `dist/` на Pages без сервера.

**Принцип розбивки:** деплой закриваємо першим (пустишка на Pages), потім фізика з тестами (кістяк «фізика правильна»), потім UI поверх готового рушія, і аж потім найважче — canvas-редактор. Кожен етап лишає застосунок запускабельним.

**Наскрізне правило:** розрахунковий рушій (`src/engine/`) не знає нічого про Svelte Flow і навіть про Svelte. Svelte Flow лише редагує граф і віддає його рушію. Це тримає фізику тестованою й переносною в CLI (NFR §5).

---

## Пропонована структура

```
napir/
  index.html
  package.json  vite.config.ts  tsconfig.json  vitest.config.ts
  src/
    main.ts
    App.svelte
    engine/                    # чистий TS, нуль залежностей від UI
      types.ts                 # Node, Pipe, Graph, SourceType, Fixture
      presets.ts               # матеріали труб (Ø, C), пресети приладів
      headloss.ts              # Хазен–Вільямс + Дарсі–Вейсбах (оракул)
      graph.ts                 # шлях джерело→кран, валідація «дерево/одне джерело»
      compute.ts               # staticLift, velocity, verdict, forward/reverse
      *.test.ts                # Vitest
    ui/
      stores.ts                # стан графа, режим, активний кран
      ResultPanel.svelte  ModeTabs.svelte  InspectorPanel.svelte
      SchematicEditor.svelte   # обгортка Svelte Flow
      nodes/ SourceNode.svelte  TeeNode.svelte  TapNode.svelte
  .github/workflows/deploy.yml
```

## Етап 0 — Скелет + деплой наскрізь

**Мета:** пустий застосунок збирається і сам публікується на Pages. Закриваємо ризик деплою до того, як є що втрачати.

- `npm create vite` → Svelte + TS. Vitest. Prettier/ESLint (легко).

- `.github/workflows/deploy.yml`: build → publish `dist/` (actions/deploy-pages).
- Один тривіальний тест, щоб CI ганяв Vitest.

**DoD:** відкривається `https://<user>.github.io/napir/`, показує заглушку. CI зелений. **Перевірка:** реальний пуш → сторінка жива.

## Етап 1 — Розрахунковий рушій (чистий TS + тести)

**Мета:** уся фізика ТЗ §3, покрита тестами, без жодного UI. Це кістяк проєкту.

- `types.ts`: `Node {id, kind: source|tee|tap, z, ...}`, `Pipe {id, from, to, L, dInner, C, kFittings}`, `Graph {nodes, pipes}`, `SourceType`, `Fixture`.
- `presets.ts`: перенести пресети труб з v0.1 (ПНД/ППР/сталь → Ø, C); пресети приладів (§4.1: душ 1.5 бар/8 л/хв тощо).
- `headloss.ts`:
  - `hazenWilliams(L_eff, D, C, Q)` — двигун (перенести з v0.1, додати `k_fittings`).
  - `darcyWeisbach(L_eff, D, roughness, Q, T)` — **оракул для тестів** (ν(T), коеф. тертя Swamee–Jain).
- `graph.ts`: `pathToTap(graph, tapId)` (єдиний шлях у дереві), `validateTree(graph)` (одне джерело, без циклів).
- `compute.ts`: `staticLift` через висоти вузлів (§3.2), `velocity` (§3.4), `verdict`, `computeForward`/`computeReverse` уздовж шляху.
- **Тести (Vitest):**
  - Золотий сценарій §8 → очікувані бари (зафіксувати числа).
  - HW ↔ DW на 2–3 сценаріях — розбіжність у межах ~5–10% (§5 «фізика»).
  - Крайові: від'ємний тиск, ліміт всмоктування >7 м лише для станції, швидкість >2 м/с.
  - Валідація графа: цикл і два джерела відхиляються.

**DoD:** `npm test` зелений, рушій рахує §8. **Перевірка:** цифри §8 звірити вручну з v0.1 (мають збігтися до `k_fittings`).

## Етап 2 — UI-оболонка й результат (без canvas)

**Мета:** робочий калькулятор поверх рушія, але граф поки — фіксований дефолтний ланцюг (сценарій §8). Перевіряємо зв'язку рушій↔UI до того, як брати важкий редактор.

- `stores.ts`: реактивний стан графа + режим + активний кран.
- `ModeTabs` (прямий/зворотний), вибір типу джерела, пресети приладів.
- `ResultPanel`: головна цифра, вердикт, розбивка (статичний підйом / тертя / по ділянках), смужка, попередження, підпис «оцінка при заданій витраті» (§3.3).
- `InspectorPanel`: редагування параметрів елемента з тимчасового списку (стане панеллю вибраного вузла на етапі 3).

**DoD:** застосунок відтворює функціонал v0.1 + нову модель (типи джерела, пресети, швидкість), але на новому рушії. **Перевірка:** ті самі входи → той самий результат, що в v0.1/етапі 1.

## Етап 3 — Візуальний редактор (Svelte Flow)

**Мета:** те, заради чого все — накидати схему на canvas і працювати з графом.

- Інтегрувати Svelte Flow. Кастомні вузли: `SourceNode`, `TeeNode`, `TapNode` (з іконками). Ребро = труба.
- Додати/видалити/перемістити вузол; провести/розірвати трубу.
- `InspectorPanel` прив'язати до вибраного елемента (параметри труби / висота вузла).
- Валідація «дерево + одне джерело» (§4.0) на льоту, з підказкою при порушенні.
- Клік на кран → активний кран → рушій рахує шлях → `ResultPanel` оновлюється наживо.

**DoD:** повний цикл — намалював схему, вибрав кран, побачив тиск. **Перевірка:** зібрати §8 у редакторі вручну → результат збігається з етапом 1.

## Ризики й відкриті питання

- **R1 — canvas на телефоні.** Вузловий редактор пальцем незручний, а ТЗ хоче мобільний («розбір у полі біля свердловини»). Реальний польовий сценарій — це *читати результат і правити числа*, а не будувати схему. Пропозиція: мобільний = перегляд + правка параметрів + результат; побудова графа — desktop-first. Узгодити на етапі 4 (або раніше).
- **R2 — `k_fittings` за замовчуванням.** Число впливає на результат, а взяте «зі стелі». На етапі 1 закласти консервативні дефолти (0.2–0.3) і показувати їх у UI як явний, редагований параметр, а не приховану константу.
- **R3 — межа застосовності HW.** Якщо юзер введе екзотику (дуже мала витрата / товста труба), HW може поїхати. Тест HW↔DW (етап 1) — це й є детектор: якщо колись розійдуться >10%, це сигнал переглянути двигун.
